# Git 协同

## 什么是版本控制

写论文或代码时，常会遇到这样的需求：改了一大段想回到昨天的版本、想知道某段话是谁什么时候加的、多人同时编辑同一份文件不想互相覆盖。Git 就是解决这些问题的工具，它按时间线记录文件的每一次变更，可以随时查看历史、对比差异、回退到任意版本。

## 核心概念

对初学者而言，Git 的核心可以理解为三个区域和一次快照。

**三个区域**

| 区域 | 说明 | 类比 |
|------|------|------|
| 工作目录（Working Directory） | 你正在编辑的文件，就是磁盘上看到的文件夹 | 书桌，你正在写写画画的地方 |
| 暂存区（Staging Area） | 你挑选出来准备记录的一组修改 | 便签，把确定要存档的改动挑出来 |
| 仓库（Repository） | Git 存储的所有历史版本 | 档案柜，所有提交过的版本都在里面 |

一次典型的操作流程：在"书桌"上修改文件 → 把想要的改动放到"便签"（`git add`，术语叫暂存）→ 把便签内容存进"档案柜"（`git commit`，术语叫提交）。每次提交都会生成一个带编号的快照，将来可以随时翻出来对比或恢复。

**提交（Commit）**

提交是 Git 的基本单位，相当于把当前所有暂存的文件拍一张快照，附上说明文字（commit message），存入仓库。每个提交有唯一的哈希值作为标识符（如 `a1b2c3d`）。提交是不可逆的存档操作，一旦提交，快照就永久可查。

**分支（Branch）**

分支是一条独立的开发线。默认分支通常叫 `main`。新建分支相当于在某个时间点分叉出一条新的时间线，两条线可以各自发展，互不干扰，需要时再合并（merge）到一起。这也是多人协作的基础，每个人在自己的分支上工作，完成后合并回主线。

## 配置 Git

使用前需配置身份信息，每次提交都会记录这些信息：

```powershell
git config --global user.email "you@example.com"
git config --global user.name "Your Name"
```

## 忽略文件

建议在项目中创建 `.gitignore` 文件，排除不需要追踪的临时文件和输出：

```gitignore
__pycache__/
*.pyc
.venv/
figures/
.DS_Store
```

## 关联远程仓库

如需远程协作，在 GitHub 等平台创建仓库后关联到本地：

```powershell
git remote add origin "https://github.com/用户名/仓库名.git"
```

## Claude Code 中的 Git 操作

在 Claude Code 中，不需要记忆命令，用自然语言即可完成 Git 操作。AI 会自动读取变更内容并生成有意义的提交信息。

| 自然语言指令 | Claude Code 实际执行 | 说明 |
|-------------|-------------------|------|
| "帮我提交更改" | `git add` + `git commit` | AI 读取所有变更，自动总结并生成 commit message，然后提交 |
| "推送 Git" | `git push` | 将本地提交推送到远程仓库 |
| "拉取 Git" | `git pull` | 从远程拉取最新更改 |
| "创建 Git 分支 xxx" | `git branch` + `git switch` | 创建并切换到新分支 |
| "合并分支 xxx" | `git merge` | 将指定分支合并到当前分支 |
| "查看改了哪些文件" | `git status` / `git diff` | AI 用自然语言解读当前变更状态 |
| "查看最近的提交历史" | `git log` | AI 总结最近的提交记录 |

VSCode 内置了 Git 可视化工具。源代码管理面板可以直观看到哪些文件有修改（绿色/红色标记），点击文件即可并排对比新旧代码差异，左下角的分支按钮能快速切换和创建分支，时间线视图可以浏览提交历史。用 VSCode 的可视化界面查看变更，更直观高效。

## 用 AI 回退变更

回退是版本控制最有价值的场景之一。改出了问题，需要回到之前的某个状态。Git 回退机制涉及多个概念（reset、revert、checkout、restore），命令行参数复杂，容易因选错操作丢失代码。在 Claude Code 中，只需描述需求，AI 会判断该用哪种方式。

### 回退未提交的修改

如果只是改了文件但还没有提交（还在工作目录或暂存区），直接告诉 AI 即可：

- "放弃所有还没提交的修改，回到上次提交的状态" → AI 执行 `git checkout -- .` 或 `git restore .`
- "只放弃 xxx 文件的修改" → AI 只恢复指定文件
- "把暂存区的文件撤回到工作目录（取消 add，但保留修改）" → AI 执行 `git reset HEAD <file>`

### 回退已提交的修改

如果已经提交了，需要根据情况选择方式：

**撤销最近一次提交，保留修改内容（推荐）**

说："撤销上一次提交，但保留这些修改，我改好了再提交"。AI 会执行 `git reset --soft HEAD~1`——提交被撤销，但文件修改回到暂存区，可以继续编辑后重新提交。

**撤销最近一次提交，也放弃修改**

说："撤回上一次提交，那些修改也不要了"。AI 会执行 `git reset --hard HEAD~1`——注意这会永久丢弃上一次提交的所有修改。

**创建一个反向提交来抵消某次提交**

说："把某次提交的效果反转掉"。AI 会执行 `git revert <commit>`——生成一个新的提交，其效果与指定提交完全相反。这比 reset 更安全，适用于已推送到远程的提交。

### 回退到任意历史版本

如果想回到几天前或某次特定提交时的状态：

- "回退到三天前的状态" → AI 会查找历史记录，确认是否是哪个提交，然后执行回退
- "回退到提交 a1b2c3d" → 直接指定提交哈希值

### 恢复误删的文件

- "恢复被我删掉的 xxx 文件，回到上次提交的版本" → AI 从 Git 历史中恢复
- "不小心删了几个文件，帮我找回来"

### 操作建议

- 如果不确定，可以先用"查看改了什么"让 AI 展示变更内容，再决定要不要回退
- 回退前建议先提交当前状态（哪怕是临时的），这样无论如何变化都还能再找回来
- 已推送到远程的提交，优先用 revert 而非 reset，避免影响协作者

## 协作工作流

在 Agent 辅助开发的模式下，Git 的角色更加关键。AI 让代码的编写从手动挡变成了自动挡。然而"代码能跑"不等于架构健康。频繁的小步提交、审查每一次 AI 变更、出问题时果断回滚，这套机制是安全使用 AI 编程的基础。把 Git 用好，就是把人机信任边界控制好。
