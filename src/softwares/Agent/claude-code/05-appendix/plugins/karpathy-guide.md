# andrej-karpathy-skills 使用指南

Andrej Karpathy 在大量使用 AI 编程后总结了 LLM 最常见的四类错误，并提出对应原则。安装后在编码、重构和调试时自动激活，也可手动执行 `/karpathy-guidelines` 显式调用。

AI 编程的核心矛盾：AI 写得快，但快不等于对。速度放大了一切——写得快的另一边是错得快、改得多、代码膨胀得快。这四条原则本质上是给速度加约束。

## 安装

```text
/plugin marketplace add forrestchang/andrej-karpathy-skills
/plugin install andrej-karpathy-skills@karpathy-skills
```

> Windows：`$env:HTTPS_PROXY=http://127.0.0.1:7890`

## Think Before Coding — 先想再写

**问题**：AI 倾向于接受你的描述并直接动手，即使你的描述有歧义或存在更简单的解法。结果方向错了，后面全是沉没成本。

**原则**：动手前先陈述它理解了什么、做了哪些假设。如果存在多种解读，列出选项让你选。如果发现更简单的实现路径，主动提出。

**反面示例**：

> 用户：帮我把用户登录改成手机号登录
>
> AI 直接改了 auth 模块，结果用户本意是"增加手机号登录方式，保留邮箱登录"。

**正确行为**：AI 会先确认——是替换还是增加？是否需要短信验证？旧用户如何迁移？确认完再动手。

**触发**：任何需要实现或修改代码的任务。

## Simplicity First — 简洁优先

**问题**：AI 天然倾向过度工程——为一个只用一次的逻辑建三个抽象层，为不可能发生的状态写错误处理，为"未来可能需要"加配置项。这些代码不会让系统更健壮，只会让后续阅读和维护更吃力。

**原则**：只写满足需求的最小代码量。判定标准——"一个有经验的工程师看了会不会说这是过度设计？"

**反面示例**：

> 用户让 AI 从 CSV 里读两列数据画散点图。AI 写了一个 DataLoader 抽象类、一个 ScatterPlotConfig 配置类、一个 FigureExporter 接口，总共 300 行。
>
> 实际需要：8 行 pandas + matplotlib。

**正确行为**：先写 8 行。如果后续真需要处理 10 种文件格式、5 种图表类型，再抽象不迟。

**关键检查点**：

- 只为用了一次的逻辑建了抽象？删掉
- 加了需求没提到的功能？删掉
- 为不可能发生的错误写了 handler？删掉
- 加了"方便以后扩展"的参数？删掉

**触发**：实现新功能、生成代码、添加模块。

## Surgical Changes — 精准修改

**问题**：AI 倾向于顺手"优化"沿途看到的其他代码——格式化、重命名、重构不相关的函数。这导致 diff 膨胀、review 困难、引入意外 bug。

**原则**：只改目标代码。不改相邻行的格式。不改"看起来可以更好"的无关函数。改了你的代码导致某行 import 无效，那就删掉那个 import——但之前的死代码不碰。

**反面示例**：

> 用户让 AI 修复 data_loader.py 里一个日期解析错误。AI 修了 bug，还顺手把整个文件的变量命名风格改了、把 for 循环重构为列表推导、加了类型注解。
>
> Diff 从 3 行变成 80 行，review 的人找不到修的是哪一行。

**正确行为**：修日期解析，3 行改动，提交。如果命名风格确实需要改，另开一个 commit。

**判定标准**：diff 里的每一行改动，能否直接追溯到你的需求？不能就删。

**触发**：编辑已有文件、修 bug、重构、代码审查。

## Goal-Driven Execution — 目标驱动

**问题**：模糊的需求产生模糊的产出。"优化一下这段代码""让页面更好看"——AI 可能改了一堆东西，但没有客观标准判断是否完成，只能靠人肉检查。

**原则**：把每个任务翻译成可验证的通过条件。AI 改完之后自己跑验证，不通过的继续改，直到满足条件。人不应该替 AI 做它能自动检查的事。

**反面示例**：

> 用户：修复这个 bug
>
> AI 改了一行代码，说"应该修好了"。Bug 还在，因为 AI 没跑测试验证。

**正确行为**：先写一个能复现 bug 的测试 → 确认测试失败 → 修改代码 → 确认测试通过。每一步有客观证据。

**不同类型任务的验证策略**：

| 任务 | 弱标准（不用） | 强标准（用这个） |
|------|-------------|---------------|
| 修 bug | 看起来修好了 | 复现测试失败 → 修改 → 测试通过 |
| 重构 | 代码干净了 | 重构前后测试全绿 |
| 加功能 | 功能写完了 | 写测试 → 测试失败 → 实现 → 测试通过 |
| 画图 | 图出来了 | 核查数据范围、轴标签、配色与规范一致 |
| 数据处理 | 跑通了 | 核查输出行数、特征温度单调性、归一化范围 |

**触发**：执行多步骤任务、调试、重构、数据处理。

## 四原则的配合

这四条不是独立的检查项，而是一张网：

1. **Think** 确认方向——别跑了半天发现跑道不对
2. **Simplicity** 约束产出——只写确实需要的，减少后面要 review 的量
3. **Surgical** 约束范围——改了 A 就别碰 B，出问题知道哪里找
4. **Goal** 定义终点——没有客观通过标准就不算做完
