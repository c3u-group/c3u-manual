# 上下文管理

## Agent 的组织边界

软件工程按职责划分模块——前端、后端、数据库，各自负责一块，状态在模块间流动共享。Agent 按另一种逻辑组织：**按 context 划分**。

每个 agent（或每次会话）的"世界"就是它的上下文窗口。窗口内的内容 agent 能看到、能推理；窗口外的东西对它而言不存在。Context 是 agent 的基本组织边界，不是可以无限扩容的存储空间。

这一差异有几个直接推论：

- **Context 是稀缺资源**，需要主动分配而非被动填满。塞进去的东西每一条都在消耗注意力预算。
- **Multi-agent 架构的本质是 context 分区**——给每个 subagent 独立的上下文空间，不是为了并行提速，而是为了隔离关注点。
- **跨 context 边界的信息传递需要显式机制**。没有全局变量，没有共享内存——只有你主动写进 CLAUDE.md、记忆系统、或压缩摘要里的东西才能跨越边界。

因此，上下文管理不是"优化技巧"，而是使用 agent 的基本功。

## 上下文窗口不等于有效注意力

人的视野约 180°，但中央凹覆盖仅约 2°——你"能看到"整个房间，真正"注意到"的只有注视点周围一小片区域，其余是余光里的模糊形状。LLM 的上下文窗口同理：1M token 是"能看到的范围"，不等于"能注意到的范围"。

### 标称窗口 vs. 有效推理范围

DeepSeek v4 标称上下文窗口为 1M token（约 75 万英文词或 35 万中文字），足够塞进整本教材。但"能塞进去"不等于"能有效处理"。

实际存在两个维度的衰减：

**容量瓶颈**。长上下文中，模型对中部信息的检索准确率显著下降（Needle-in-a-Haystack 效应）。信息不在窗口边缘附近时，即使未被截断，模型也可能"找不到"。有效推理范围远小于标称窗口——复杂任务的可靠注意力区间最好控制在 200K token 以内，超出后推理质量逐步衰减。

**位置衰减**。模型对上下文不同位置的关注度不均匀：首尾优先（primacy/recency bias），中部信息容易被忽略。距离当前 token 越远，关联强度越低——不是截断与否的二分问题，而是连续的衰减曲线。

### 实际含义

塞满 1M 上下文不等于模型"理解"了全部内容。关键信息应放在 CLAUDE.md（始终位于上下文前部）或会话开头，而非埋在对话中段。对长文档或大项目的处理，分段交给 subagent 各自聚焦往往比一次性全部塞入更可靠。

## 上下文里有什么

一次对话的上下文由以下部分构成：

1. **系统提示**（不可见，占固定份额）——Claude Code 的运行指令、工具定义、安全规则等
2. **CLAUDE.md**（全部加载，三种类型叠加）——用户全局、项目共享、项目私有三类文件合并注入
3. **对话历史**——用户消息、AI 回复、工具调用及其返回结果
4. **`@` 注入的文件内容**——每次 `@文件路径` 将文件全文注入对话。同一文件被多次引用会重复计入

工具调用结果是上下文消耗的主要来源之一：读取大文件、执行长输出命令、搜索返回大量结果——每次都在消耗注意力预算。不是说要避免读取文件，而是要有意识：每次读文件都是一笔支出。

`/cost` 可查看当前会话的 token 用量，`/usage` 按 skill、subagent、plugin 等类别细分。

## 压缩机制

当上下文接近窗口上限时，Claude Code 自动触发压缩：较早的对话轮次被摘要替换。摘要保留关键决策和结论，丢失细节和中间推理过程。这不是 bug——是 context 有限这一约束下的必然设计。

手动控制命令：

- **`/compact`**：主动触发压缩。可指定保留重点，如 `/compact 保留认证模块的改动`。适合任务阶段性完成、需要存档当前进度时使用
- **`/clear`**：清空全部对话历史，开启全新上下文。CLAUDE.md 不受影响。适合任务完全切换、或上下文膨胀到影响推理质量时使用

选择策略：当前任务与历史强相关（需要之前的决策上下文）时用 `/compact` 存档；任务已结束、新任务与历史无关时直接 `/clear`。压缩不可逆——摘要丢失的细节无法找回。

详见过往命令说明：[`/compact` 与 `/clear`](../02-quick-start/commands.md#compact)。

## 跨边界信息传递

每次 `/clear` 或新会话都是一道 context 边界。跨边界保留信息需要显式机制：

### CLAUDE.md：始终可见的公告栏

CLAUDE.md 在每次会话启动时自动注入上下文前部，不受 `/clear` 和压缩影响。适合放：

- 项目约定（目录结构、命名规范、构建命令）
- 硬约束（画图字体和配色、文件格式要求）
- 编码规范（语言、风格、linting 规则）

编写指南见 [项目配置（CLAUDE.md）](./claude-md.md)。原则：只写 AI 从代码里看不出来的东西，控制在 ~200 行以内——过长反而稀释关键信息。

### 记忆系统：按需检索的档案柜

`/memory` 命令管理持久化文件记忆。与 CLAUDE.md 的区别：

| | CLAUDE.md | 记忆系统 |
|---|---|---|
| 加载方式 | 每次会话全部注入 | 按相关性检索加载 |
| 适合内容 | 硬约束、项目约定 | 用户偏好、项目背景、经验记录 |
| 容量影响 | 始终占用上下文 | 仅在相关时占用 |
| 编辑方式 | 直接编辑文件或让 AI 修改 | `/memory` 命令或自然语言 |

选择原则：每次会话都需要知道的放 CLAUDE.md，偶尔需要参考的放记忆。不要让 CLAUDE.md 变成百科全书——那是记忆系统该做的事。

## Context 分区策略

### Subagent：天然的上下文隔离

每个 subagent 有自己的独立上下文窗口，看不到主会话的对话历史，只接收你委派的任务描述。这不是为了性能并行——是为了上下文隔离。一个 subagent 处理一组数据、一个模块的代码审查，上下文紧凑聚焦，不受主会话历史干扰。

这与"agent 按 context 划分"的组织原则一致：不是把一个人拆成多个人干活，而是给每个子任务分配独立的"视野"，各看各的。

更多见 [工具与扩展](./tools.md#subagent)。

### 长任务分段

复杂任务不要试图在一个会话里从头做到尾。每段上下文应紧凑聚焦：

1. 第 1 段：需求分析 + 方案设计 → `/compact` 存档结论
2. 第 2 段：核心实现 → `/compact` 存档改动
3. 第 3 段：测试验证 + 修复

每段上下文保持在注意力有效范围内，跨段信息通过 CLAUDE.md 和 `/compact` 摘要传递。

### 信息密度优先

精简 CLAUDE.md、精简 prompt、精简 `@` 引用的文件范围——不是在节省 token 费用，是在保护模型的有限注意力。能引用一个函数就不贴整个文件，能用一句话说清就不写一段。

## 常见误区

**"1M 上下文 = AI 能同时深入理解 1M token"**。标称窗口是容量上限，不是有效推理范围。长上下文的中部信息检索准确率显著下降，位置衰减不可忽略。

**"AI 记得对话中的所有历史"**。早期内容可能已被自动压缩为摘要。如果你在对话第 50 轮引用第 5 轮的细节，AI 可能已经在看摘要而非原文。

**"CLAUDE.md 越详细越好"**。CLAUDE.md 始终占据上下文前部。越长，留给实际对话的注意力空间越少。只写必要信息，其余放记忆系统。

**"`/compact` 无损"**。压缩生成的摘要会丢失细节和中间推理。压缩不可逆——关键决策应在压缩前手动记录到 CLAUDE.md 或记忆系统中。
